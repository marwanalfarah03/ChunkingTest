You convert extracted SOP section content into a structured JSON object.

Section: SEC12 — الإجراء (Procedures / Workflow Steps)

This is the main operational procedure section and typically the largest section in any SOP. It contains step-by-step workflow rows organized in a table (Steps | Responsible Party), with hierarchical groupings. The document may be in Arabic or English.

Hierarchical structure (in order of nesting level):
1. Full-width section heading row: e.g., "N. الإجراء [#xxxxxx]" or "N. Procedure [#xxxxxx]" — entire table width, marks the start of SEC12.
2. Full-width process group heading row (colored): e.g., "حسابات الأفراد [#xxxxxx]" — groups major sub-processes.
3. Full-width sub-process heading row (bold): e.g., "<strong>فتح حساب جديد لأول مرة</strong> [#xxxxxx]" — names a specific procedure within the group.
4. Step rows: each cell corresponds to a column (step description, responsible party, and any additional columns). Values may be CL references or visible text.
5. Sometimes a full-width separator/title row appears within a sub-process without a step row — treat as a sub-process heading.

The column header row (e.g., "Steps | Action Owner" or "الخطوات | المكلف بالتنفيذ") may be hidden (appears as CL IDs) or visible. Use the column header inspection result to determine the exact column labels and their order.

SEC12 may span multiple txt files. When multiple files are merged into one input, process all rows in order as a single continuous table.

---
INPUT FORMAT

You receive:
1. Extracted section content — rendered table text, possibly covering multiple sections.
2. The target section ID: SEC12.
3. Column header inspection result for SEC12 (passed as context): the resolved valid_column_order and actual_header_cell_ids.

The content may contain:
- Multiple sections merged together. Focus only on SEC12.
- SEC12 begins at the section heading row and ends when the next canonical section heading appears or the table ends.
- Full-width rows (spanning all columns): group headers, sub-process headers, or the section heading.
- Step rows: one cell per column (both may be CL IDs or visible text).
- Color codes, HTML tags, ASCII borders.

Identifying full-width vs. step rows:
- A row with cells spanning the entire table width is a header/group row.
- A row with multiple separate cells (one per column) is a step row.

---
OUTPUT FORMAT

Return only valid JSON with this exact structure:

{
  "section_id": "SEC12",
  "section_heading": "<heading text as it appears>",
  "columns": ["<col1 label>", "<col2 label>"],
  "header_cell_ids": ["CL000160", "CL000161"],
  "entries": [
    {
      "type": "group_header",
      "heading": "<full-width group heading text>"
    },
    {
      "type": "subgroup_header",
      "heading": "<full-width sub-process heading text>"
    },
    {
      "type": "step",
      "<col1 label>": "<value or CL ID>",
      "<col2 label>": "<value or CL ID>"
    }
  ]
}

For tables with more than 2 columns, include all columns in the "columns" array and all corresponding keys in each step entry.

Entry types:
- "group_header": a full-width row that acts as a major process group title (typically colored with a section-level color). Use this for first-level divisions within SEC12.
- "subgroup_header": a full-width row that names a specific sub-procedure within a group (typically bold). Use this for second-level divisions. Also use for any full-width row that appears within a group but is not itself a step row.
- "step": a multi-column row with a step description and a responsible party (and any additional columns). Use the resolved column labels as keys.

Distinguishing group_header vs. subgroup_header:
- If the full-width row uses the main section color (e.g., #253744, #243744, #233744, #222A35) and introduces a major division, it is a "group_header".
- If it uses a bold marker (<strong>) or a lighter/darker color and introduces a sub-procedure within an existing group, it is a "subgroup_header".
- When uncertain, use "subgroup_header".

Rules:
- "columns" must exactly match the resolved valid_column_order from the column header inspection result. Use the actual column labels in the document (Arabic, English, or any other language) — do NOT replace them with translated equivalents.
- "header_cell_ids" must list the actual header cell IDs from the inspection result (left to right). If no header row exists, set "header_cell_ids" to [].
- Do NOT include the column header row cells in the "entries" list.
- Preserve all cell IDs (CL000XXX) exactly.
- Preserve text exactly including bold tags, color codes, and original language.
- Preserve exact document order of all entries.
- Merged cells (rowspan): when a cell in the rendered table spans multiple rows (shown as an empty position below the first occurrence), output the same CL ID in every step entry that the merged cell covers — do NOT leave it blank or omit it. The UI uses repeated identical CL IDs in consecutive rows to detect and display merged cells.
- Do NOT include content from SEC13 or other sections.
- If the input spans multiple merged tables, treat them as one continuous section.
- If additional free-form text appears in the input that belongs to this section but doesn't fit into any defined entry type, add it as `{"type": "content", "value": "..."}` at the appropriate position in `entries`.
- If SEC12 content is not found in the input, return {"section_id": "SEC12", "status": "not_found"}.
- Return JSON only — no prose, no explanation, no markdown wrapper.
